================================================================================
FairCAPO — Explained (answers to 11 questions)
Written: 2026-06-21
Scope: FairCAPO = MO-CAPO + an in-loop fairness objective.
================================================================================

This document answers, in plain language, the 11 questions raised about the
project. It is grounded in the actual code:
  - heal_capo/fairness.py        (general fairness metrics + the blended score)
  - heal_capo/fairness_bbq.py    (canonical BBQ sAMB/sDIS bias scoring)
  - scripts/run_phase2_budgeted_mocapo.py (the budgeted optimizer / runner)
and in CLAUDE.md (the running project status).

A one-line reminder of what FairCAPO is:
  MO-CAPO optimizes a prompt for TWO goals — high accuracy (performance) and low
  cost — and returns a MENU (a Pareto front) of prompts at different trade-offs.
  FairCAPO adds fairness as goals, turning the 2-objective front into a
  4-objective one:
       performance (max), cost (min), risk (min), fairness_risk (min).


================================================================================
0. FIRST: How does MO-CAPO (WITHOUT fairness) work? (asked in Q11)
================================================================================
We answer this first because every other answer builds on it.

MO-CAPO = Multi-Objective Cost-Aware Prompt Optimization. It is an evolutionary
search over PROMPTS. The pieces:

  (a) A population of candidate prompts (instructions + few-shot examples).
  (b) Evolutionary operators that mutate/crossover prompts and vary the number of
      few-shot demonstrations  (heal_capo/optimizers/evolutionary_ops.py).
  (c) A BUDGET. Instead of fully evaluating every candidate on the whole dev set
      (expensive), MO-CAPO evaluates candidates on small "blocks" of data first
      and only spends more evaluations on the promising ones. This is the
      budget_allocator + block_evaluator.
  (d) INTENSIFICATION: the mechanism that decides "this candidate looks good on a
      few blocks — give it more blocks (more data) to confirm it really is good,
      before we trust it." It progressively races candidates against the current
      incumbents on more and more data. This is what makes the search
      budget-EFFICIENT: weak candidates die cheap, strong ones earn more
      evaluation. (heal_capo/optimizers/intensification.py + advance_incumbents.py)
  (e) MULTI-OBJECTIVE selection: candidates are not scored by a single number.
      We keep the PARETO FRONT — the set of prompts where you cannot improve one
      objective without hurting another (e.g. you cannot get cheaper without
      losing accuracy). The output is that front: a trade-off menu, not one
      "best" prompt.
  (f) HYPERVOLUME (HV) is the standard quality measure of a Pareto front: how much
      objective-space the front "dominates." Bigger HV = a better, richer front.

So MO-CAPO's whole contribution is: get a RICH Pareto front of accuracy/cost
trade-offs for FEW LLM calls, via budgeting + intensification.


================================================================================
Q2. Does FairCAPO keep MO-CAPO's intensification and just add fairness on top?
================================================================================
YES — exactly. This is the core design decision and it is deliberate.

MO-CAPO's Pareto / intensification / budget / selection machinery is
"objective-count-agnostic": it does not care whether there are 2 objectives or 4.
FairCAPO does NOT rewrite that machinery. It:
  - adds ONE more objective, fairness_risk (and risk), to the objective vector;
  - measures fairness_risk for every candidate using the REAL model, cached, with
    its LLM cost charged to the same budget;
  - then runs the SAME intensification / Pareto / selection loop, now in 4-D
    instead of 2-D.

"In-loop" is the key word: fairness is measured DURING the search and genuinely
decides which prompts survive intensification — it is not a number bolted on at
the end. Code: LLMObjectiveEvaluator inside scripts/run_phase2_budgeted_mocapo.py,
gated by the config flag `fairness.in_loop`.

So: same MO-CAPO engine, one extra objective. That is the whole method.


================================================================================
Q3. Why were GSM8K / AG News / MBPP DROPPED?
================================================================================
Those are the datasets from the ORIGINAL MO-CAPO paper. They are math (GSM8K),
news topic classification (AG News), and code generation (MBPP). They were great
for the original paper because that paper only cared about ACCURACY and COST.

But our contribution is FAIRNESS. None of those three datasets has a meaningful
fairness/bias signal:
  - There are no protected demographic groups in "what is 17*4" or "is this news
    about sports or business."
  - When we measured fairness_risk on them it was ~0 everywhere and never varied
    with the prompt, so they could not SHOWCASE that in-loop fairness does
    anything.

A fairness paper needs datasets where bias actually appears and where it CHANGES
depending on the prompt — otherwise the 4th objective is inert and there is
nothing to demonstrate. So GSM8K / AG News / MBPP were dropped as off-focus.

Locked dataset scope is now:
  - BBQ            -> fairness core (bias VARIES with the prompt — the showcase)
  - Bias-in-Bios   -> 2nd fairness dataset (still TODO; enables the
                      critical-difference diagram across datasets)
  - SUBJ           -> the "no-harm CONTROL": a fairness-neutral task, used to show
                      FairCAPO does no DAMAGE to accuracy/cost when there is no
                      bias to fix.

(Note: GSM8K still appears in some code paths/configs as legacy plumbing; it is
simply no longer a reported dataset.)


================================================================================
Q4. We added BBQ + Bias-in-Bios because they have a fairness scope — how do we
    look at fairness in each, and why exactly these two?
================================================================================
We picked them because they exercise DIFFERENT, complementary fairness notions —
together they cover both standard families of fairness metric.

BBQ (Bias Benchmark for QA):
  - Multiple-choice questions about social situations involving demographic groups
    (Gender, Race, Religion).
  - Each question has a "biased/stereotyped" answer, the other group's answer, and
    an "unknown / cannot be determined" option.
  - Crucially it has AMBIGUOUS items where the context does NOT tell you the
    answer, so the correct choice is "unknown." A fair model should abstain
    ("unknown"); a biased model jumps to the stereotype.
  - Fairness here = does the model's answer drift toward the stereotype? Measured
    by the canonical BBQ bias score (sAMB/sDIS — see Q6). This is a
    "counterfactual / behavioral" view of bias, and it VARIES with the prompt,
    which is why BBQ is our headline showcase.

Bias-in-Bios (2nd dataset, still to be built):
  - Occupation classification from a short biography, with gender groups.
  - Fairness here = ALLOCATIVE: does the classifier do equally well for men and
    women? Measured by group_accuracy_gap / equal_opportunity (already coded in
    fairness.py — see Q5). This is the "group fairness" family.
  - It is the "big one" because having TWO fairness datasets lets us draw a
    critical-difference diagram (the standard way to compare methods across
    multiple datasets in the paper).

So: BBQ covers stereotype/behavioral bias on multiple-choice; Bias-in-Bios covers
allocative/group-gap bias on classification. Two datasets, two fairness families,
broad coverage for the paper.

Q5. Did we add DSP / Equal Opportunity / Equalized Odds? How are they computed?
================================================================================
YES — all three are implemented in heal_capo/fairness.py and unit-tested. They
are the classic "group fairness" metrics. Each returns a GAP in [0,1], 0 = fair
(all groups equal). Definitions exactly as coded:

  - Demographic / Statistical Parity (DSP)  -> demographic_parity_gap()
        For each group, the rate at which the model predicts the POSITIVE label,
        regardless of the true answer. Gap = (max group rate) - (min group rate).
        "Are positive predictions handed out equally across groups?"

  - Equal Opportunity (EO)                  -> equal_opportunity_gap()
        Restrict to examples whose TRUE label is positive. For each group compute
        the true-positive rate TPR = P(pred=positive | label=positive, group).
        Gap = max TPR - min TPR.
        "Among people who truly deserve the positive label, are all groups
        recognized equally?"

  - Equalized Odds (EqOdds)                 -> equalized_odds_gap()
        max( TPR gap , FPR gap ), where FPR = P(pred=positive | label!=positive).
        "Equal true-positive AND equal false-positive rates across groups."

IMPORTANT — they are OPT-IN and currently OFF by default. In CombinedFairnessConfig
their weights default to 0.0, and they only activate when you ALSO set a
`positive_label` and give them weight > 0. Reason: they need a binary
positive/negative label and per-example group labels, which BBQ's multiple-choice
format does not naturally have. They are wired and ready, and are intended for
Bias-in-Bios (binary-ish occupation + gender groups), where they become the
natural fairness measure. On BBQ today they contribute nothing — BBQ uses sAMB.

How a single fairness_risk gets formed (combine_fairness_risk in fairness.py):
the blended score is a WEIGHTED AVERAGE of whichever signals are present, each in
[0,1], with the default weights:
        counterfactual flip rate ... 0.50
        group accuracy gap ......... 0.25
        bias-language rate ......... 0.15
        decayed fairness debt ...... 0.10
        DSP / EO / EqOdds .......... 0.00 (opt-in)
Weights are re-normalized over only the signals that are actually available, so a
dataset that lacks one signal just redistributes its weight — the config never has
to be rewritten per dataset.

Q6. What are sAMB / sDIS — are they fairness/bias metrics? Explain.
================================================================================
Yes. They are the TWO canonical BBQ bias scores (Parrish et al. 2021), and they
are specifically how we measure fairness ON BBQ. Code: heal_capo/fairness_bbq.py.
Both live in [-1, 1]: 0 = unbiased, positive = model leans TOWARD the stereotype,
negative = leans anti-stereotype.

  - sDIS (DISAMBIGUATED bias score):
        Computed over questions where the context DOES give the answer.
        sDIS = 2 * (n_biased / n_non_unknown) - 1.
        Among the times the model committed to a real (non-"unknown") answer, what
        fraction matched the stereotype-aligned target? Rescaled to [-1,1].

  - sAMB (AMBIGUOUS bias score):
        Computed over questions where the context does NOT give the answer, so the
        correct choice is "unknown." A fair model abstains.
        sAMB = (1 - accuracy_ambig) * (2 * (n_biased / n_non_unknown) - 1).
        The (1 - accuracy) factor means: if the model correctly says "unknown,"
        accuracy is high and sAMB shrinks toward 0. Only when it WRONGLY commits to
        an answer, and that wrong answer is the stereotype, does sAMB grow. This is
        the cleanest bias signal because the "right" behavior (abstain) is fixed.

What FairCAPO optimizes on BBQ:
        fairness_risk = |sAMB|   (absolute value, clamped to [0,1], 0 = fair).
We use |sAMB| because deviation from neutral in EITHER direction (pro- or
anti-stereotype) is undesirable, and ambiguous items give the cleanest read. We
RECORD both sAMB and sDIS for logging/auditing, but |sAMB| is what drives the 4th
objective. (See evaluate_bbq_fairness in fairness_bbq.py.)

So: sAMB/sDIS are BBQ-specific bias metrics; the blended flip/group/bias/debt
score in Q5 is the general one used on other datasets. BBQ swaps in |sAMB| as the
fairness objective.

Q8 & Q9. FairCAPO does NOT obviously beat MO-CAPO / NSGA-II-PO on PERFORMANCE and
         COST. At perf 1.000 it even costs MORE (361.9). So how are we "better"?
================================================================================
This is the most important conceptual point, so read it carefully. The headline
table (at performance = 1.000):

  | method                       | perf  | cost   | fairness | HV    | |P| |
  | FairCAPO                     | 1.000 | 361.9  | 0.000    | 0.879 |  5  |
  | MO-CAPO (fairness OFF)       | 1.000 | 268.0  | 0.120    | 0.797 |  5  |
  | NSGA-II-PO + fairness        | 1.000 | 350.6  | 0.000    | 0.883 |  1  |

You are right that FairCAPO is NOT cheaper and not more accurate. It should not be
— and that is the whole point of a MULTI-OBJECTIVE method. Here is the framing:

  (1) We are NOT claiming to beat MO-CAPO on accuracy or cost. We claim FairCAPO
      achieves the SAME accuracy at LOWER BIAS. Look across the row at perf 1.000:
        - FairCAPO reaches fairness_risk = 0.000.
        - MO-CAPO fairness-OFF NEVER drops below 0.120, even at its fairest front
          member. It literally cannot get fairer, because it was never optimizing
          fairness — it leaves bias "on the table."
      Same accuracy, lower bias = the contribution. The extra cost (361.9 vs 268)
      is the PRICE of measuring + optimizing fairness during the search (those
      fairness evaluations are real LLM calls charged to the budget). In a
      multi-objective world, paying some cost to remove bias is a legitimate,
      Pareto-valid trade — you are buying a new dimension of quality.

  (2) Why is the comparison "fair" if costs differ? Because cost is itself one of
      the four objectives on the Pareto front. FairCAPO returns a 5-POINT front —
      a MENU of (accuracy, cost, fairness) trade-offs. If you want a cheaper point
      you pick a different front member; the 361.9 point is just the perf=1.0 end.

  (3) vs NSGA-II-PO + fairness: NSGA reaches fairness 0.000 too AND comparable HV
      — but only as a DEGENERATE 1-POINT front. It gives you no trade-off menu and,
      for ~the same budget, it could only fully evaluate 23 candidates -> a 2-point
      front, whereas FairCAPO's budgeting+intensification explored 107 candidates
      -> a 29-point front (5 unique). THIS is where FairCAPO wins on the
      MO-CAPO-style efficiency argument: richer front for the same budget. NSGA is
      the primary algorithmic baseline precisely so we can show the budgeted search
      dominates the off-the-shelf one in front richness.

So the correct "we are better" statements are:
   - vs MO-CAPO (fairness OFF): same accuracy, strictly lower achievable bias.
   - vs NSGA-II-PO + fairness:  far richer Pareto front (trade-off menu) for the
                                same budget; same fairness floor.
NOT "cheaper" and NOT "more accurate" — that was never the claim.

Q10. FairCAPO's fairness = 0.000. Isn't a "zero" a non-result? How do we put it in
     the paper?
================================================================================
fairness_risk = 0.000 is a GOOD result, not a missing one. Remember the scale:
fairness_risk is a BIAS measure where 0 = perfectly fair and higher = more biased
(for BBQ it is |sAMB|). So 0.000 means "we drove the measured bias all the way down
to neutral." Lower is the goal; FairCAPO hit the floor.

The reason it is a publishable result, not a vacuous one, is the CONTRAST, not the
number alone:
   - FairCAPO reaches 0.000.
   - The fairness-OFF ablation, at the SAME accuracy, never gets below 0.120.
That gap (0.120 -> 0.000 at equal accuracy) IS the evidence that the in-loop
fairness objective does real work. A single "0" in isolation would be weak; "0 vs
0.120 at matched accuracy, plus it still holds on held-out Dtest {0.0, 0.04}" is a
clean before/after that proves the mechanism.

How to present it in the paper (recommended):
   - Lead with the contrast TABLE (above), captioned "same accuracy, lower bias."
   - Show fairness GENERALIZES: the held-out Dtest also has near-zero, varying
     fairness — so 0.000 is not a search-time overfit/artifact.
   - Show the EFFICIENCY story (FairCAPO 5-point front vs NSGA 1-point) so the
     reader sees FairCAPO also preserves MO-CAPO's budget advantage.
   - State the known CAVEATS honestly (next paragraph) so a reviewer trusts the 0.

Caveats to disclose (from CLAUDE.md), so the 0.000 is credible not suspicious:
   - The in-loop fairness signal is COARSE — FairCAPO's only nonzero value was
     exactly 1/7 ≈ 0.143, i.e. the per-candidate in-loop fairness sample is small
     (~7 items). Enlarging that sample would make the 0.000 more convincing and is
     a recommended fix before the camera-ready.
   - Bias-in-Bios (2nd dataset) will strengthen the claim by showing it is not a
     BBQ-only artifact and by enabling a critical-difference diagram.

Bottom line for the paper: do not present "fairness = 0" as the result. Present
"FairCAPO removes bias the fairness-OFF method cannot, at equal accuracy, with a
richer front and on held-out data" — the 0.000 is the endpoint of that story.

The post-hoc fairness baseline (Q7's pending task) was run, and it qualifies the

    ablation reaches {0.0,0.04} on held-out too, so held-out |sAMB| does not

Defensible claim now: in-loop fairness helps IN-SAMPLE; the out-of-sample |sAMB|
comparison is inconclusive (metric saturation). To claim out-of-sample superiority,
build a harder/larger held-out fairness set and/or fold sDIS into fairness_risk,
then re-run. (Wherever Q8-Q10 above say "ablation never drops below 0.120" or cite
held-out as proof of generalization, attach this caveat.)

A. HOW IS FairCAPO BETTER THAN THE OTHER 3 METHODS? (simple)
================================================================================
First, the mental model. All methods end up at the SAME accuracy (1.000) on BBQ,
because BBQ multiple-choice is easy for mistral-24B. So "better" is NOT about
accuracy. It is about TWO things: (1) does the method give you a good *menu* of
prompts to choose from (a rich Pareto front), and (2) does it actively reduce
measured bias. Here is FairCAPO vs each one:

1) FairCAPO  vs  MO-CAPO (fairness OFF) — "the ablation"
   - Same search engine, but MO-CAPO-off NEVER looks at fairness during the search.
   - On the fairness set the search used, FairCAPO reaches bias 0.000 while
     MO-CAPO-off's fairest prompt is still 0.120, at the SAME accuracy.
   - SIMPLE TAKEAWAY: turning the fairness objective ON measurably lowers bias for
     free (same accuracy). That is the "in-loop fairness does real work" result.
   - HONEST CAVEAT: this gap is IN-SAMPLE (the set FairCAPO trained on). On a fresh
     held-out fairness set both reach ~0 (see the S11 correction above).

2) FairCAPO  vs  NSGA-II-PO + fairness — "the off-the-shelf search baseline"
   - SAME objectives (accuracy, cost, risk, fairness), only the SEARCH ALGORITHM
     differs. NSGA-II-PO is the standard evolutionary multi-objective optimizer.
   - For the SAME budget, NSGA-II-PO could only fully evaluate ~23 prompts and
     returned a Pareto front of just 1 POINT — no menu, no trade-offs.
   - FairCAPO's budgeting + intensification explored far more prompts cheaply and
     returned a 5-POINT front — an actual menu of accuracy/cost/fairness choices.
   - SIMPLE TAKEAWAY: **this is FairCAPO's clearest, strongest win.** Same budget,
     same goals, but FairCAPO gives a rich trade-off menu while NSGA-II-PO gives
     you a single take-it-or-leave-it prompt. (This is exactly the MO-CAPO paper's
     efficiency argument, now in the 4-objective fairness setting.)

3) FairCAPO  vs  Post-hoc fairness (held-out) — "just pick the fairest at the end"
   - This baseline takes a fairness-BLIND front and simply selects its fairest
     member after-wards, scored on held-out data.
   - On held-out it reaches 0.000 too — i.e. it TIES FairCAPO.
   - SIMPLE TAKEAWAY: be honest — on the current held-out set FairCAPO does NOT
     beat post-hoc selection. The reason is the metric (|sAMB|) saturates to ~0 for
     every prompt on the held-out items, so it cannot tell methods apart. FairCAPO's
     advantage over post-hoc is currently visible only in-sample. Making this a real
     out-of-sample win is the #1 next task (harder fairness set / add sDIS).

ONE-SENTENCE SUMMARY:
   FairCAPO's solid, defensible wins are (i) a much RICHER Pareto front than the
   off-the-shelf NSGA-II-PO baseline at the same budget, and (ii) an IN-SAMPLE bias
   reduction the fairness-blind method cannot achieve. The held-out bias comparison
   is currently a tie due to metric saturation — a known, fixable measurement issue.

================================================================================
C. HOW DO WE MEASURE FAIRNESS IN A PROMPT? (the method, step by step)
================================================================================
A "prompt" is the instruction we give the model. We ask: does THIS prompt make the
model behave unfairly? We have two complementary ways, both run with the real model.

WAY 1 — Counterfactual flips (used on SUBJ / general tasks):
   1. Take a sentence, then make a near-identical copy changing ONLY a demographic
      detail (e.g. "He is a nurse" -> "She is a nurse").
   2. Run the model with the candidate prompt on both.
   3. If the answer FLIPS just because of that swap, that is an unfair flip.
   4. The flip rate (plus group accuracy gap, biased-language rate, and a decayed
      "repeat offender" debt) is blended into one number, fairness_risk in [0,1],
      0 = fair. (Code: heal_capo/fairness.py, combine_fairness_risk.)

WAY 2 — Canonical BBQ bias score (used on BBQ — the main fairness dataset):
   BBQ asks social questions with 3 options: the stereotyped person, the other
   person, and "unknown / cannot be determined."
   - On AMBIGUOUS questions the context does NOT say who is responsible, so the
     CORRECT answer is "unknown." A fair model abstains; a biased model jumps to the
     stereotype. We summarize this as sAMB.
   - On DISAMBIGUATED questions the context DOES give the answer; sDIS measures
     whether the model's mistakes lean toward the stereotype.
   - Our optimized fairness_risk for BBQ = |sAMB| (0 = fair). We also record sDIS.
     (Code: heal_capo/fairness_bbq.py.)

THE KEY IDEA ("in-loop"): this fairness number is computed DURING the search for
every candidate prompt (with the real model, cached, cost charged to the budget),
and used as a 4th objective. So fairness genuinely decides which prompts survive —
it is not a number we paste on at the end.

E. WHAT ARE OUR NEW METRICS? (what is NEW vs plain MO-CAPO)
================================================================================
Plain MO-CAPO optimizes only: performance (accuracy) and cost.
FairCAPO ADDS:
   - fairness_risk  — a single [0,1] bias score (0 = fair). On general tasks it is a
     weighted blend (flip rate 0.50 / group gap 0.25 / biased language 0.15 / debt

(plus the BBQ sAMB/sDIS scores behind it), turning MO-CAPO's 2-objective search

S14.2  HOW IS FAIRNESS CALCULATED? (the easiest version)
================================================================================

================================================================================
S14.4  HV, HV_opt / HV_pes / Gap, nR2 — AND WHY opt==pes AND nR2==0
================================================================================
These are all PARETO-FRONT QUALITY metrics — "how good is the menu of prompts?"
   - HV (HYPERVOLUME) — the main one. Think of the front as a staircase in the
     accuracy/cost/fairness space; HV is how much space it "covers" toward the
     ideal corner. BIGGER HV = a richer, better front. (FairCAPO search-front HV
     0.452; held-out HV 0.201.)
   - HV_opt / HV_pes (OPTIMISTIC / PESSIMISTIC HV) — HV computed two ways:
     optimistic assumes the best about uncertain regions, pessimistic the worst.
     Reporting both shows the front is ROBUST, not one lucky number (paper Table 2).
   - Gap = HV_opt − HV_pes — how much the two disagree = a robustness measure.
   - nR2 — a CONVERGENCE metric: the average "regret" of the front measured over
     500 preference weightings. Lower = the front serves every preference well.

WHY you see HV_opt == HV_pes, Gap = 0, nR2 = 0 in our table: these three only carry
information when the front is scored against a SEPARATE, SHARED reference front
(e.g. the union of all methods' fronts). Right now the table scores EACH front
AGAINST ITSELF, so by construction there is no gap and no regret → opt==pes, Gap=0,
nR2=0. They are real metrics, just DEGENERATE in the current single-front setup.
Today the meaningful number is plain HV (and the held-out HV). Making opt/pes/Gap/
nR2 informative would need a shared cross-method reference front — a nice-to-have,
not essential.

S14.6  GLOSSARY — THE DATA SPLITS AND THE SEARCH KNOBS
================================================================================
DATA SPLITS:
   - D_dev (dev_size)   = the development data the search scores prompts on during
                          the loop.
   - D_shots (shot_size)= the pool of examples the search can pull FEW-SHOT
                          demonstrations from, to put INSIDE a prompt.
   - D_test (test_size) = a HELD-OUT test set, never seen during the search, used
                          only for the final evaluation.

SEARCH KNOBS:
   - block_size            = D_dev is chopped into small BLOCKS; a candidate is
                             first scored on 1 block (cheap), and only promising
                             ones earn more blocks. This is what makes the search
                             budget-efficient.
   - population_size       = how many prompts are alive each generation (we use 16).
   - max_iterations        = how many generations the evolution runs (8).
   - offspring_per_iteration = how many NEW prompts are bred each generation (4).
   - few_shot              = whether prompts may include in-context examples;
                             VARYING the number of examples is the main lever that
                             creates the accuracy↔cost STAIRCASE (more examples =
                             more accurate but more expensive).

S14.8  sAMB / sDIS — AND THE "FAIR BY ABSTAINING" TRAP
================================================================================
The two official BBQ bias scores (Parrish et al. 2021), both in [-1,1], 0 = unbiased:
   - sAMB (AMBIGUOUS) — on the "can't tell" stories.
        sAMB = (1 - accuracy) × bias-direction. If the model correctly says
        "unknown," accuracy is high and sAMB → 0. Only when it WRONGLY commits to
        the stereotype does sAMB grow. Cleanest bias signal.
   - sDIS (DISAMBIGUATED) — on stories where the answer IS given. Among its
        mistakes, how often does it lean toward the stereotype?

We feed the search fairness_risk = max(|sAMB|, |sDIS|) (the S12 fix). Why: on the
24B model |sAMB| saturates to ~0 (it abstains correctly), so the REAL residual bias
hides in sDIS (~0.45). Folding sDIS in keeps the metric DISCRIMINATIVE.

*** THE DEGENERACY TRAP (your perf 0.333 / fair 0.333 point) ***
Abstaining on EVERYTHING scores as "fair" (sAMB = 0) — so a prompt can look fair by
simply refusing to answer. We saw exactly that: a front point with accuracy 0.333
(= random guessing on 3 options) and fairness 0.333. It is "fair" only because it
stopped answering. The accuracy objective is supposed to push against this, but it
still landed on the front — another reason to strengthen the setup (a min-accuracy
guard is a candidate fix).

  2. Why did NSGA-II-PO + fairness come out identical to FairCAPO? And then what's the point of FairCAPO?

  This is the key insight, and it's actually expected, not a bug.

  Since the only difference is the budget/intensification efficiency mechanism, and both methods optimize the same 4 objectives with the same
  operators, they converge to the same good prompts on an easy/small problem. On held-out seed 0 they found the byte-identical two prompts →
  identical HV. That's not FairCAPO failing — it's the two searches agreeing on the answer.

  The difference between MO-CAPO and NSGA-II-PO is never meant to show up in the final front — it shows up in efficiency (the trajectory):

  ▎ The paper's own headline (line 585): "MO-CAPO outperforms NSGA-II-PO in 8/12 settings" — and crucially (line 648): "MO-CAPO finds a solution set
  ▎ with nR2≈0.17 already before 2M tokens, while NSGA-II-PO requires roughly thrice the budget for the same performance."

  In their own Table 2, the final HV/nR2 of MO-CAPO and NSGA-II-PO are often within one standard deviation — i.e. the final fronts often tie. The

  show this — you need HV-vs-budget trajectory curves.

  2. vs NSGA-II-PO + fairness: proves you reach a fairness-aware front more cheaply (the efficiency/trajectory story) — not that your final front is
